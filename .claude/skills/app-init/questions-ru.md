# Канонические формулировки вопросов (RU)

Используй эти тексты дословно в `AskUserQuestion`. У каждого вопроса — короткий заголовок
(`header`, ≤12 символов), у каждого варианта — человеческое описание последствия. Ровно один
вариант помечай как рекомендованный (ставь его первым и добавляй «(рекомендуется)»).

## Round 1 — фиксированные вопросы

### Q1 — header «Тип»
Вопрос: «Что мы создаём?»
- **Сайт-визитка / лендинг** — Красивые страницы и формы. Без личных кабинетов и базы данных. Запускается мгновенно.
- **Приложение с данными** — Люди входят и сохраняют данные: записи, заявки, товары, клиенты.
- **Пока не знаю** — Помогите выбрать (задам пару уточняющих вопросов).

### Q2 — header «Доступ»
Вопрос: «Нужен ли вход по логину?»
- **Без входа** — Любой посетитель пользуется сразу, ничего не регистрируя.
- **Личные кабинеты** — Каждый регистрируется и видит только свои данные.
- **Один администратор** — Вход только для вас; посетители просто смотрят.

### Q3 — header «Интеграции» (multiSelect: true)
Вопрос: «Что приложение должно уметь дополнительно? (можно выбрать несколько)»
- **Отправка писем на e-mail** — Подтверждения и уведомления на почту.
- **Уведомления в Telegram** — Сообщения вам в Телеграм о новых событиях.
- **Приём оплаты** — Кнопка оплаты картой.
- **Ничего из этого (рекомендуется)** — Только основное. Добавить можно позже.

### Q4 — header «Данные» (только для пути «Приложение с данными»)
Вопрос: «Кто видит данные?»
- **Каждый — только свои (рекомендуется)** — Стандартно и безопасно.
- **Все видят всё** — Общая лента или каталог, доступный всем.
- **Только администратор** — Данные видны лишь вам.

### Мини-уточнение (если выбрали «Пока не знаю»)
- «Нужно ли сохранять данные между визитами?» → Да · Нет
- «Будут ли люди входить под своим аккаунтом?» → Да · Нет
Любое «Да» → приложение с данными; иначе → сайт-визитка.

## Round 2 — примеры вопросов под идею (генерируй похожие)

Задавай вопрос, только если ответ реально меняет таблицу/экран/сценарий и его нельзя
безопасно угадать. 2–4 вопроса, не больше.

### Пример A — личный блог («блог про путешествия», статика, один админ)
- «Как будете добавлять статьи?» → Через меня в чате (рек.) · Простая админка с паролем · Файлами Markdown
- «Что показываем на главной?» → Лента последних статей (рек.) · Большое фото + 3 статьи · Плитки категорий
- «Нужны ли комментарии?» → Нет (рек.) · Да, с модерацией · Да, через Telegram

### Пример B — бронирование столиков («сайт бронирования», БД)
- «Что бронируют?» → Столик на время (рек.) · Конкретный столик по схеме зала · Просто заявка «перезвоните мне»
- «Длина брони?» → Слот 1.5–2 часа (рек.) · Фиксированно на весь вечер · Гость указывает сам
- «Часы работы?» → Каждый день 12:00–23:00 (рек.) · Разное по дням · Настрою потом
- «Подтверждение брони?» → Автоматически (рек.) · Вы подтверждаете вручную

### Пример C — CRM («crm для агентства», БД, личные кабинеты)
- «Главные сущности?» (multiSelect) → Клиенты + Сделки (рек.) · Задачи · Заметки
- «Как двигаются сделки?» → Канбан-доска по этапам (рек.) · Список со статусом · Просто список
- «Командная работа?» → Только я (рек.) · Несколько сотрудников, общие клиенты · Каждый видит своих
- «Что на старте?» → Пустая база (рек.) · Заполнить демо-данными для примера
